1.	Introduction
 
1.1	Purpose
This document describes the functional and non-functional requirements for an Ayurvedic Doctor Channeling Platform consisting of:
Patient (Customer) Side - Web and Android applications
Doctor Side - Web and Android applications
Superadmin Side - Web console
The platform will be available as:
Web Application (for all roles)
This SRS is intended for product owners, designers, developers, testers, and stakeholders to understand the system requirements and guide the development process.
1.2	Scope
1.2.1	Patient (Customer) Capabilities
The system will allow patients to:
Discover Ayurvedic doctors and clinics
View doctor profiles, specializations, languages, locations, and consultation types (online, in-person)
View available time slots and make bookings
Pay consultation fees online or mark as pay-at-clinic
Receive reminders, access prescriptions and basic medical history
Provide ratings and feedback
Manage appointments (cancel, reschedule)
Join online consultations via video/audio
View medical records and prescription history
1.2.2	Doctor Capabilities
The system will allow doctors to:
Create and manage their profiles and clinic locations
Define consulting schedules and fees
Accept or reject bookings
Conduct online consultations through integrated video or audio
Record visit notes and prescriptions
Track earnings and payouts
Manage appointment calendar
View patient history
1.2.3	Superadmin Capabilities
The superadmin will:
Onboard and verify doctors and clinics
Manage specializations, treatments, content and configurations
Handle disputes and refunds (outside payment flow, unless integrated)
View analytics and export reports
Manage system settings and master data
Monitor system performance
1.2.4	Out of Scope
The following features are explicitly out of scope for Version 1.0: Better if can integrate
•	Back office tools like accounting systems
•	Call center dashboards 
•	Marketing automation iOS application (planned for future versions) 
•	In-app chat between patient and doctor
•	Loyalty points and referral programs

2.	Overall Description
 
2.1	Product Perspective
The Ayurvedic Doctor Channeling Platform is a standalone system that integrates with:
Payment Gateways - For processing online payments
SMS Gateways - For OTP and notifications
Email Services - For account and booking communications
Video Conferencing Platforms - For online consultations
Maps Services - For location and navigation features
2.2	User Roles
The system supports three primary user roles:
2.2.1	Patient (Customer)
End users seeking Ayurvedic consultations
Can browse doctors, book appointments, and manage their health records
Access via web portal or Android mobile app
2.2.2	Doctor
Licensed Ayurvedic practitioners
Manage profiles, schedules, appointments, and patient records
Access via web portal or Android mobile app
Must be verified by superadmin before activation
2.2.3	Superadmin
Platform administrators
Manage doctors, system configuration, and oversight
Access via web console only
Full system access for support and compliance
2.3	Platforms
2.3.1	Web Application
Patient Web Portal - Full-featured web interface for patients
Doctor Web Portal - Full-featured web interface for doctors
Superadmin Web Console - Administrative dashboard and management tools
2.3.2	Backend API
Both web and mobile platforms will share the same RESTful backend API, ensuring consistency across all platforms.
2.4	Product Functions
The system provides the following major functions:
1.	User Authentication & Authorization - Secure login, registration, and role-based access
2.	Doctor Discovery - Search, filter, and browse Ayurvedic doctors
3.	Appointment Management - Booking, cancellation, rescheduling
4.	Payment Processing - Online payment and pay-at-clinic options
5.	Online Consultations - Video/audio consultations
6.	Prescription Management - Digital prescriptions and medical records
7.	Notifications - SMS, email, and in-app notifications
8.	Reviews & Ratings - Patient feedback system
9.	Admin Management - Doctor verification, system configuration, reporting
2.5	User Characteristics
2.5.1	Patients
Varying levels of technical proficiency
May prefer mobile or web based on convenience
Need simple, intuitive interface
Require multilingual support (Sinhala, Tamil, English)
2.5.2	Doctors
Moderate to high technical proficiency
Need efficient tools for managing appointments and patients
May prefer web for detailed work, mobile for quick access
Require professional interface
2.5.3	Superadmin
High technical proficiency
Need comprehensive administrative tools
Require detailed reporting and analytics
2.6	Constraints
1.	Regulatory Compliance - Must comply with local healthcare data protection laws
2.	Internet Connectivity - Online consultations require stable internet
3.	Payment Gateway - Dependent on third-party payment service availability (koko payments)
4.	Device Compatibility - Android app must support Android 6.0+ (API level 23+)
5.	Browser Support - Web app must support modern browsers (Chrome, Firefox, Safari, Edge)
2.7	Assumptions and Dependencies
2.7.1	Assumptions
Patients and doctors have stable internet connections for online consultations
Local laws regarding storage of medical information and Ayurvedic prescriptions will be observed
Users have access to smartphones or computers
Payment gateway integration will support local currency and cards, and optionally mobile wallets SMS and email gateways are available for OTP and notifications
2.7.2	Dependencies
  Payment gateway service provider

4.	Functional Requirements
 
Throughout this document, requirement identifiers use the format: REQ_<MODULE>_<NN>
4.1	Common: Authentication & User Management
4.1.1	Sign Up and Login
Patients
  REQ_AUTH_01: The system shall allow a patient to create an account using:
Mobile number Email address   Social login (Google, Apple)
  REQ_AUTH_02: The system shall validate mobile number or email using:
OTP (One-Time Password) sent via SMS or email
Email verification link
  REQ_AUTH_03: The system shall allow login using:
Email or mobile number plus password
"Continue with Google" option
"Continue with Apple" option (on supported devices)
Doctors
  REQ_AUTH_04: The system shall allow doctors to create accounts, but keep them in "Pending verification" status until approved by superadmin.
Superadmin
  REQ_AUTH_05: Superadmin users shall only be created and managed from the admin console. No public registration for superadmin role.
4.1.2	Password Management
  REQ_AUTH_06: The system shall provide "Forgot password" functionality with:
OTP sent to registered mobile/email
Password reset link sent via email
Enforcement of strong password rules:
Minimum 8 characters
At least one uppercase letter
At least one lowercase letter
At least one number
At least one special character
  REQ_AUTH_07: After password reset, the user shall be able to log in with the new password only. Old password shall be invalidated.
4.1.3	Session Management
  REQ_AUTH_08: The system shall maintain user sessions with configurable timeout (default: 30 minutes of inactivity).
  REQ_AUTH_09: The system shall allow users to log out from all devices.
  REQ_AUTH_10: The system shall support "Remember Me" functionality for extended sessions.
4.1.4	Profile Types and Onboarding
Patients
  REQ_PAT_01: The system shall collect the following patient information during registration:
Full name (required)
Mobile number (required, verified)
Email (optional if mobile used)
Gender (optional)
Date of birth (required)
Preferred language(s) (required, at least one)
  REQ_PAT_02: The patient shall be able to update profile details except:
Verified mobile number (requires re-verification)
Verified email (requires re-verification)
Doctors
  REQ_DOC_01: The system shall collect the following doctor information during registration:
Full name (required)
Clinic or practice name (required)
Registration number with Ayurvedic regulatory body (required) Specializations (should be in a dropdown)  (required, at least one):
Panchakarma
Pediatrics
General Medicine
Women's Health
Geriatrics
Others (customizable by admin)
Years of experience (required)
Languages spoken (required, at least one)
Consultation types (required, at least one):
In-person
Online
  Home visits
Per-consultation fees and currencies for each location/type (required)
Bank details for payouts (required for approval)
  REQ_DOC_02: The system shall require upload of supporting documents for verification:
Registration certificate
National Identity Card (NIC) or equivalent
Professional photo
Additional documents as required by regulatory body
